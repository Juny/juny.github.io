本篇文章主要针对有一定的服务端开发或者测试经验的同学，以及对被分析的应用有一定的了解，或者有相关的开发负责人员辅助。
主要内容：

  1. 常见性能问题以及如何避免
  2. 常见性能问题的分析方法以及工具的使用
  3. 实际案例分析

2. 常见性能问题以及分析方法

常见性能问题

  • 三高
    CPU，内存，线程数
  • 慢
    业务响应时间慢
  • 和预期结果不对
    看似理论上没问题的代码，结果就是不对
  • 资源不释放
    资源不释放导致的内存溢出，系统资源耗尽，响应慢
    资源：文件、链接、端口、句柄……
  • IO
    网络IO，磁盘IO
  • 其他

CPU高

CPU高有可能本身不是问题，有可能就是运算需求比较高。
如果CPU资源有限，可以尽可能的优化，使CPU在可能的范围内降到最低。

可能造成CPU高的情况 :

  • 死循环
  • 没必要的操作（本地IP、初始化资源。。）
  • 频繁线程切换
  • 不正常的使用某些类（map）
  • 频繁FGC

可能导致CPU高的情况

  1. 代码Bug
    可能性太多……
  2. 死循环
    手抖写的死循环或者计算失误导致死循环
  3. 线程数太多（线程数量是否比预期的异常的高）
    大量的线程切换
    频繁创建线程
    线程池不正常的使用（FixedThreadPool,CacheThreadPool）
  4. FGC
    频繁的FGC也会导致CPU高
  5. 其他
    大量数据的复制和一些类的不正常使用，比如map的resize，list的clone
    没必要的操作（本地IP、初始化资源。。）

分析方法

  • 用到的工具和命令
    jvisualvm
    jstack，jstat，jmap，ps，top

  • 情况一：单核CPU使用率一直100%
    大多情况为死循环或者某个线程一直在执行大量运算操作

  • 情况二：单核CPU使用率不定时100%
    应用本身有周期性的任务
    非人为的周期性操作，比如FGC,大量数据的copy（list,map）

  • 情况三：每核CPU使用率都高
    需要结合堆栈、代码、资源占用、内存情况等，全面的考虑和分析。

Java CPU 问题分析

  • 方法1
    Java Visualvm直接连接进程分析，能够比较直观的看到每个方法所用的CPU时间，更容易定位问题。大多数情况下生产环境无法使用。适用于被测对象问题能够复现，且配置的远程rmi监控的情况。生产环境不会配置远程监控。

    java服务启动时需配置启动参数：
    -Dcom.sun.management.jmxremote.port=8999 -Dcom.sun.management.jmxremote.ssl=false -Dcom.sun.management.jmxremote.authenticate=false -Djava.rmi.server.hostname=192.168.143.136

    使用方法见demo演示。

  • 方法2
    在CPU高的时候抓线程堆栈，适用于CPU相对比较高的情况，如果条件允许可以把服务压力加大到尽量大，使CPU尽量高后去分析问题。

    ps -mp pid -o THREAD,tid,time | awk ‘{printf $2” “$8”\n” }’ | sort

    printf “%x\n” tid

    jstack pid | grep –A 20 tid
    `

    连续抓取几次堆栈信息，分析正在运行的线程，分析当前操作是否耗费CPU，这么多运行的是否正常，业务堆栈是否正常，该业务可能出现的问题等等。
